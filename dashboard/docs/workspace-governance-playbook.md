# Workspace 治理运行手册（V1）

适用范围：`croupier-dashboard` + `croupier` 的 Workspace 配置发布与回滚链路。  
目标：固化灰度、回滚、环境隔离规则，降低线上配置变更风险。

## 1. 灰度发布策略（TASK-043）

### 1.1 灰度对象定义

- `objectKey`：按单对象灰度，不做全局一刀切发布。
- `role`：先对 `admin/tech_lead/tester` 开放，观察后再放量。
- `tenant/scope`：有游戏服或业务域隔离时，优先按 scope 灰度。

### 1.2 灰度规则定义

- 阶段 1（验证）：仅内部角色可见，持续 30 分钟无 P1/P0 异常再进入阶段 2。
- 阶段 2（小流量）：目标业务 10%-20% 对象启用，持续 2 小时观察。
- 阶段 3（全量）：全角色全对象放开。
- 触发回退条件：
  - `workspace_render_error` 持续出现。
  - 发布后关键页面空白/不可操作。
  - 回归清单中核心链路失败（保存/发布/控制台访问/回滚）。

## 2. 回滚 SOP（TASK-044）

### 2.1 标准回滚步骤

1. 在 `WorkspaceEditor` 打开目标 `objectKey` 的“版本历史”。
2. 选择可用历史版本，先看“版本详情 + Diff”。
3. 执行“回滚”并确认风险提示。
4. 回滚后立刻验证控制台详情页可访问。
5. 必要时执行重新发布，让控制台切到回滚后的稳定版本。

### 2.2 事故处理流程

1. 发现异常后 5 分钟内先止血：冻结发布、定位对象、明确影响范围。
2. 15 分钟内完成回滚并二次验证。
3. 30 分钟内输出事故记录：时间线、根因、影响范围、恢复动作、预防项。
4. 对同类对象补充回归用例与预检查规则。

## 3. 环境隔离与配置校验（TASK-045）

### 3.1 环境规则

- `dev`：允许频繁导入导出和模板试验；不作为发布质量基线。
- `test`：回归与联调环境；必须走完整发布/回滚链路验证。
- `pre`：预发布验收环境；配置冻结窗口内仅允许修复性变更。
- `prod`：生产环境；仅允许审批通过的配置发布。

### 3.2 环境差异校验

- 校验项：
  - `objectKey` 是否一致。
  - `layout` 顶层与 tab 子布局是否在 V1 支持范围。
  - 必需函数是否在目标环境存在。
  - 权限点是否匹配目标角色策略。
- 建议执行顺序：
  1. 先导出 `metadata` 与 `backup bundle`。
  2. 在目标环境导入前执行 `validateWorkspaceConfig`。
  3. 发布前查看“变更摘要”，确认差异在预期范围内。
