# Workspace Product Delivery Guide

适用范围：历史对象工作台交付方案记录，仅保留用于迁移参考。  
目标：把“函数 -> 工作台 -> 控制台”的产品链路、能力边界、发布规则和样板使用方式讲清楚，供产品、实施、前端和 QA 统一执行。

## 1. 产品主路径

当前稳定主路径固定为三层：

1. 函数目录
2. 对象工作台
3. 运行控制台

各层职责：

- 函数目录：确认对象有哪些原子能力可用，检查 descriptor、入参、实例、路由和单函数配置。
- 对象工作台：把函数装配成页面，生成骨架，补字段与布局，做预览，执行发布前检查。
- 运行控制台：验证已发布结果，确认入口名、页面层级、真实数据态与运行可读性。

不建议的做法：

- 在函数详情里停留过久，把单函数 UI 配置当成最终业务页面。
- 在没有主函数、没有页面骨架时直接做复杂布局。
- 在编辑器预览看起来“像样”后，跳过控制台正式态验证直接交付。

## 2. 能力矩阵

### 2.1 正式能力

这些能力允许进入默认主路径、默认模板和正式交付承诺：

- 顶层布局：`tabs`
- Tab 布局：`form-detail`、`list`、`form`、`detail`
- 工作台保存 / 发布 / 取消发布 / 回滚
- 发布前检查 / 质量评分 / 发布前摘要
- 控制台正式运行态验证

### 2.2 Beta 能力

这些能力可创建、可预览、可渲染，但交付前必须额外核对：

- `kanban`
- `timeline`
- `split`
- `wizard`
- `dashboard`
- `grid`
- 多函数编排向导

Beta 能力交付要求：

- 必须走真实函数绑定验证，不能只看示例态。
- 必须在控制台正式态复核空态、错态、真实数据态。
- 必须在交付说明里明确标记为 Beta 能力。

### 2.3 实验能力

这些能力允许保留代码，但不进入默认主路径，也不作为当前对外承诺：

- `custom` / `single`
- 画布编辑器
- 节点式增强能力
- 其他未闭环实验入口

实验能力使用规则：

- 不进入默认模板推荐。
- 不作为“官方演示样板”交付基础。
- 需要显式告知使用方这是实验能力。

## 3. 页面装配最佳实践

### 3.1 起首个页面

推荐顺序：

1. 绑定主函数
2. 自动生成基础布局
3. 补最小可用字段 / 主列 / 详情分区
4. 切到预览看首屏结构
5. 再继续扩更多 Tab

不要一上来就做的事：

- 先开高级编排
- 先堆多个 Tab
- 先纠结视觉微调

### 3.2 什么时候扩更多页面

满足下面条件再扩第二个 Tab：

- 首个页面已有主函数
- 核心函数已补齐
- 预览能看懂主流程
- 发布前检查没有硬阻塞

### 3.3 什么时候切到高级模式

只有下面情况才建议进入 `TabEditor` 高级模式：

- 单页面需要多函数编排
- 标准骨架无法覆盖页面结构
- 需要补全复杂布局缺失配置
- 需要回看编排历史并做撤销 / 重做

普通列表页、表单页、详情页优先在基础模式完成。

## 4. 官方演示样板使用规则

当前官方样板至少覆盖三类：

- 运营列表台
- 对象详情台
- 查询-处理工作台

样板的用途：

- 对外演示产品能力
- 作为视觉和信息架构基线
- 作为实施团队的起始模板

样板使用要求：

- 套用后必须按模板自带 checklist 做一次核对
- 必须确认真实函数是否已经绑定
- 必须在控制台正式态验证，不只看编辑器预览

不建议：

- 直接把官方样板当成生产成品，不做对象字段和权限适配
- 把样板中的占位字段、占位动作直接发布

## 5. 发布规范

### 5.1 发布前最低门槛

必须满足：

- 至少一个可见 Tab
- 已绑定主函数
- 核心函数存在
- 关键字段 / 主列 / 详情分区已补齐
- 没有发布前硬阻塞

### 5.2 有风险提示时的处理

如果只有 warning，没有 blocking：

- 可以发布
- 但必须在发布前摘要中确认影响范围
- 必须在控制台里走一次真实验证

### 5.3 控制台验证清单

发布后至少核对：

1. 入口名是否清晰
2. 页面首屏是否能读懂
3. 当前版本是否正确
4. 真实数据态是否可信
5. 空态 / 错态是否明确

只要这五项有一项不通过，就不应认为已经完成交付。

## 6. 角色协作建议

产品 / 设计：

- 关注页面是否能读懂、主动作是否明确、信息层级是否成立。

实施 / 前端：

- 关注函数绑定、字段结构、布局完整度、发布前检查和控制台验证。

QA：

- 优先覆盖正式主路径，再补 Beta 能力回归。
